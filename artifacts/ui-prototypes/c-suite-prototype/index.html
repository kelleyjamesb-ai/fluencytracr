Agents & MCP tool health

AI agents reach FluencyTracr through a standard adapter. One path sends in metadata events only (no chat text). Three paths pull back evidence summaries: the full bundle, coverage, or controls view. Every call is logged with an outcome — it succeeded, was rejected, came back suppressed, or hit an error — so you can see if the channel is healthy and whether unsafe or malformed requests were stopped at the door.

The same Function buttons above change these demo figures. In production you would tie calls to a function when your org’s grouping labels include that slice and there is enough volume to do it safely.

How the program lands with leaders

Outside FluencyTracr, perceived value usually means what people feel they gain versus what they give up (benefits vs. hassle, cost, or worry). We use that idea loosely here — not as a formal score, but as four readings you could support with behavior evidence, coverage, and short anonymous check-ins. Wording stays humble: these describe mood and fit at org level, not who is “good” or “bad.”

Scope

We describe how the organization uses AI: verification habits, abandonment, friction patterns, and sanctioned vs unsanctioned tool signals. We also show whether agents using the MCP adapter are succeeding or hitting schema, safety, or upstream errors — still at org or function scope, not per person. We do not rank people. Small teams roll up until the group is big enough to report safely.

Can this run on real data?

Yes. The product already defines the shapes; the UI wires them together.

  • Headline tiles: roll up GET /api/observability/:orgId (executions, disclosure, pattern mix, suppression reasons) and coverage fields from evidence APIs — optionally filtered by org unit / function when your ingest tags allow and k-min is met.
  • Raw input count: sum or disclose ingest volume from your event store or a dedicated counter, with the same privacy boundary as aggregates (no raw prompts in UI).
  • Shadow AI %: aggregate “unsanctioned tool class” style signals from the evidence / indexing model — still org- or function-level, not user-level.
  • Agent / MCP channel: aggregate the MCP adapter audit stream (result: success · rejected · suppressed · error) by tool name (fluency.ingest_events, fluency.get_evidence_bundle, etc.). Ship counts to metrics or log analytics; map org_unit to function when tags support it.

Ground rules

The Deeper view section above is how these pieces look once each slice has enough volume and your pipelines match. Perceived-value-style readings mix signals you already collect with optional anonymous pulses — never a substitute for blaming a team.

Anything that would identify a person or rank employees stays out — even if the raw warehouse could compute it.

Where FluencyTracr results can show up

  • This executive view (or embedded analytics in your portal)
  • Search-indexed evidence summaries, when your org publishes bundles to its enterprise search stack
  • Governed assistants via the FluencyTracr MCP server (stdio or streamable HTTP) — same four tools, audited every call
