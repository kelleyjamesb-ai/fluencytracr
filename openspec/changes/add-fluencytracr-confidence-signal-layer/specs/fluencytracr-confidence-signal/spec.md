## ADDED Requirements
### Requirement: Task Episode Schema
The system SHALL represent each task episode with explicit identifiers, timestamps, closure reason, and ambiguity state (including status and reason code) without silent defaults.

#### Scenario: Episode recorded with ambiguity details
- **WHEN** an episode is persisted
- **THEN** it includes episode_id, function_id, role_class, start_ts, end_ts, closure_reason, and ambiguity_state with status and reason_code

### Requirement: Ambiguity Detection and Suppression
The system SHALL mark episodes as AMBIGUOUS when any ambiguity trigger is present and SHALL suppress all behavioral signals derived from ambiguous episodes.

#### Scenario: Ambiguous episode suppresses signals
- **WHEN** an episode is flagged AMBIGUOUS due to inactivity or conflicting signals
- **THEN** the episode is excluded from latency, disposition, ghost-use detection, and convergence inputs

### Requirement: Signal Class Gating
The system SHALL only allow surfacing when at least two distinct behavioral signal classes are present in the observation window.

#### Scenario: Single-class observations suppress surfacing
- **WHEN** only one signal class is observed in the window
- **THEN** the surfacing decision is SUPPRESS

### Requirement: Iteration Depth Classification
The system SHALL derive iteration depth labels (None, Light, Heavy) from follow-up turn counts relative to the function baseline and SHALL NOT store iteration depth as a persistent field.

#### Scenario: Iteration depth derived at evaluation time
- **WHEN** follow-up counts are evaluated for a function
- **THEN** the system labels depth according to the defined ranges without storing the label

### Requirement: Latency Restrictions
The system SHALL treat latency as a corroborative signal only, assign zero contribution for configured role classes, and ignore latency for ambiguous episodes.

#### Scenario: Latency excluded for ambiguous episodes
- **WHEN** an episode is AMBIGUOUS
- **THEN** latency does not contribute to any surfacing decision

### Requirement: Verification Safe Mode
The system SHALL record verification as a binary presence based on human-initiated interactions and SHALL exclude automated retries from verification.

#### Scenario: Automated retries do not count as verification
- **WHEN** a verification event is generated by automation
- **THEN** verification presence remains false

### Requirement: Confidence Gating Logic
The system SHALL apply the V1 confidence gating sequence and emit required suppression reason codes when any gate fails.

#### Scenario: Insufficient time suppresses surfacing
- **WHEN** the observation window is under 30 days
- **THEN** the system returns SUPPRESS with reason INSUFFICIENT_TIME

### Requirement: Non-Tunable Parameters
The system SHALL enforce fixed windows, binary confidence output, default SUPPRESS state, and no overrides or tunable parameters.

#### Scenario: Attempted override rejected
- **WHEN** a configuration attempts to change the observation or surfacing window
- **THEN** the system ignores the change and remains SUPPRESS by default

### Requirement: Governance Enforcement
The system SHALL enforce the ambiguity doctrine and anti-surveillance principle by preferring suppression over additional data collection and forbidding individual attribution.

#### Scenario: Data collection suppressed under ambiguity
- **WHEN** ambiguity or missing instrumentation is detected
- **THEN** the system suppresses signals without expanding data collection

### Requirement: Out-of-Scope Safeguards
The system SHALL NOT implement individual-level metrics, productivity scoring, recommendations, content inspection, tunable thresholds, or closed-loop automation.

#### Scenario: Individual scoring request rejected
- **WHEN** a request attempts to compute individual productivity scores
- **THEN** the system prevents the operation and records it as out-of-scope
